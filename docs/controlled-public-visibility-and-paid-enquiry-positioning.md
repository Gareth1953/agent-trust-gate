# Controlled Public Visibility And Paid Enquiry Positioning

P3-M128 adds controlled public visibility and paid enquiry positioning for
Agent Trust Gate.

Core rule:

No mandate. No evidence. No verified intent. No signed gate pass. No settlement.

## Purpose

This pack helps Agent Trust Gate move from a public repository with discovery
assets to a carefully visible public project with a clear, human-reviewed paid
enquiry route.

This is documentation, positioning, manual public-visibility planning, and paid
enquiry clarity only. It does not add outreach automation, scraping, contact
harvesting, forms, tracking, analytics, telemetry, paid ads, live APIs, payment
processing, settlement, hosted services, or action execution.

## Controlled Public Visibility Strategy

The intended path is:

1. A developer or reviewer discovers Agent Trust Gate.
2. They understand the trust problem.
3. They run the local proof/demo.
4. They understand what is and is not claimed.
5. They see a relevant paid-review use case.
6. They send a human-reviewed enquiry manually.
7. A paid technical review, local pilot feasibility discussion, or integration
   assessment may follow if separately agreed.

Visibility should be controlled, accurate, and review-led. The project should
be easy to inspect but hard to misread as a live product, payment service,
settlement system, security certification, or autonomous agent endpoint.

## Why Careful Visibility Matters

Careful visibility matters because Agent Trust Gate sits near high-impact
topics: agent actions, sensitive tools, human approval, payment-adjacent
workflows, settlement-like decisions, evidence, proof, and audit receipts.

Overstating the project would create risk. The project should be visible enough
for serious technical reviewers to find and inspect it, but every public message
must preserve the local-demo-only boundary.

## What Public Visibility Should Achieve

Public visibility should help a reviewer:

- understand the core trust problem;
- clone and run local demos;
- inspect schemas, receipts, proofs, adversarial cases, and integration
  patterns;
- understand buyer use cases and revenue triggers;
- see a cautious paid enquiry route;
- understand the safety boundary before contacting Gareth.

Public visibility should not attempt to automate lead generation, create
tracking funnels, collect contact details, run ads, or imply guaranteed buyer
demand.

## How Paid Enquiries Should Be Positioned

Paid enquiries should be positioned as:

`Paid technical review / local pilot feasibility / integration assessment`

This is safer than positioning Agent Trust Gate as a production product. It
matches the repo's current strengths: local proof artifacts, deterministic
tests, schemas, receipt/proof examples, adversarial cases, reference integration
patterns, and cautious paid pilot readiness.

A paid enquiry can ask for:

- Agent Workflow Trust Review;
- Pre-Settlement Trust Review;
- Human Approval and Evidence Review;
- Signed Receipt / Replay Risk Review;
- Sensitive Tool-Call Gate Review;

## What A Paid Enquiry Cannot Assume

A paid enquiry cannot assume:

- production deployment;
- payment activation;
- settlement authority;
- legal/compliance certification;
- security certification;
- procurement approval;
- automatic acceptance;
- automatic access after payment;
- hosted service access;
- live API access;
- external-agent contact;
- guaranteed commercial result.
