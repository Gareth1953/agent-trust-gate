## Purpose

This sequence defines a safe staged path for public visibility. It is manual
planning only. It does not send posts, contact anyone, scrape contacts, collect
leads, place ads, track users, call APIs, process payments, settle anything,
deploy services, or execute actions.

## Stage 1 - GitHub Metadata / Manual Repo Polish

Goal: make the public repo understandable before sharing.

Actions:

- confirm README start-here path;
- confirm clone/run route;
- confirm local CLI route;
- confirm GitHub metadata guidance;
- confirm safety boundary;
- confirm paid enquiry route;
- confirm old email is absent and `gpmiddleton71@gmail.com` is visible.

Stop if the repo could be mistaken for a production service, live API, payment
system, settlement system, or hosted enforcement product.

## Stage 2 - Manual Post To Gareth-Controlled Accounts Only

Goal: publish a small manual signal through accounts Gareth controls.

Actions:

- use manual copy only;
- link to the public repo;
- avoid production, certification, payment, settlement, guarantee, or automatic
  access claims;
- monitor whether readers misunderstand the boundary.

Stop if messaging causes confusion or readers assume production readiness.

## Stage 3 - Manual Sharing To Known/Relevant Technical Contacts Only

Goal: share with known, relevant reviewers where a trust-gate review may be on
topic.

Actions:

- share one relevant repo link;
- explain local-demo-only status;
- point to the README start-here path;
- invite feedback or a human-reviewed enquiry only if appropriate.

Stop if spam-like behaviour would be required or if the contact is not relevant.

## Stage 4 - Manual Developer/Community Post Where Rules Allow

Goal: make the project discoverable in relevant communities without spam.

Actions:

- check community rules first;
- disclose that the repo is local-demo-only;
- use a short manual post;
- avoid repeating posts;
- do not scrape or harvest contacts.

Stop if community rules do not allow project sharing, if paid ads would be
needed, or if tracking/analytics would be required.

## Stage 5 - Respond Manually To Genuine Enquiries

Goal: answer genuine questions without overclaiming.

Actions:

- explain what the project is;
- explain what it is not;
- point to local proof/demo commands;
- point to paid enquiry positioning if relevant;
- keep all responses human-reviewed.

Stop if the enquiry requests live integration before safe review, production
signing, live payment processing, settlement execution, wallet/banking logic,
external-agent contact, or action execution.

## Stage 6 - Invite Serious Enquiries To Paid Technical Review / Local Pilot Discussion

Goal: route serious, relevant enquiries to a safe next step.

Actions:

- ask for workflow type, sensitive action, desired local review scope, and
  constraints;
- keep the offer framed as paid technical review / local pilot feasibility /
  integration assessment;
- keep scope, timing, fee, deliverable, and access subject to separate human
  agreement.

Stop if the enquiry assumes automatic paid-pilot acceptance, automatic access
after payment, production readiness, live payment/settlement readiness,
legal/compliance/security certification, or guaranteed commercial result.

## Stop Conditions

Stop controlled distribution if:

- messaging causes confusion;
- people assume production readiness;
- payment/settlement claims are misunderstood;
- legal/compliance/security certification is inferred;
- spam-like behaviour would be required;
- enquiries request live integration before safe review;
- a channel requires scraping, contact harvesting, paid ads, tracking,
  analytics, telemetry, forms, or automation;
- a reviewer asks for secrets, credentials, private keys, wallet details,
  banking details, production logs, or customer data in public.
